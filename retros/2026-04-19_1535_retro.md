## KPT 회고

### Keep (좋았던 것)
- 문제 생성, 검토, validator 설계, 커밋 분리까지 한 흐름 안에서 계속 이어 붙였기 때문에 요구사항이 빠르게 구체화됐다. 특히 사용자 피드백이 바로 스킬 문서와 생성기 설계로 연결된 점이 좋았다.
- 노트북 품질 검토 단계에서는 `read_notebook`, `read_cell`, `insert_cell`, `overwrite_cell_source` 같은 Jupyter 계열 도구를 적극적으로 써서 셀 단위 수정 근거를 확보했다. 텍스트 파일처럼 노트북을 억지로 다루지 않은 점이 안정적이었다.
- 문서 변경, 생성기 코드, 실행 환경 변경을 커밋 그룹으로 분리하려는 시도가 반복됐다. 이 방향은 이력 관리와 회귀 추적에 유리하다.

### Problem (나빴던 것)
- `python`과 `docker`를 실제 환경에서 바로 쓸 수 있다는 가정이 여러 세션에서 반복적으로 깨졌다. 같은 종류의 실패를 다시 밟으면서 시간과 토큰이 소모됐다.
- 현재 구조에서 제거된 경로(`scripts/aice_generator/specs`)나 없는 파일(`.gitignore`)을 계속 참조했다. 코드/문서/운영 규칙이 동시에 갱신되지 않아 생긴 드리프트다.
- Windows PowerShell 환경에서 경로 글롭, UTF-8 인코딩, Git 잠금 파일, Docker 권한 같은 운영성 이슈가 자주 끼어들었다. 작업 본질보다 실행기 레벨의 마찰이 컸다.
- Jupyter RTC, BOM 포함 노트북, 데이터 다운로드 실패처럼 생성 파이프라인 앞단의 전제조건이 불안정했다. 생성 자체보다 환경 복구에 더 많은 시도가 들어갔다.

### Try (개선해야 할 것)
- 생성기와 validator 앞에 공통 preflight를 두고 `python`, `docker`, 네트워크, 대상 디렉터리, Git 잠금 상태를 먼저 점검하자. 충족되지 않으면 즉시 중단하고 구체적 조치만 안내해야 한다.
- 스킬 문서, README, validator가 참조하는 경로와 실행 방식을 단일 소스로 맞추자. 제거된 디렉터리나 Docker-first 원칙 위반을 잡는 smoke check를 추가하면 재발을 줄일 수 있다.
- Windows 기준 명령 템플릿을 정리하자. UTF-8 읽기, 경로 quoting, `rg` 패턴, Git staging, Docker 호출을 표준화하면 도구 레벨 오류를 크게 줄일 수 있다.
- 노트북 생성 직후에 JSON 파싱, BOM 검사, 필수 파일 존재 여부를 먼저 확인하고 Jupyter RTC 접근은 그 다음에 하자. 순서를 바꾸면 후속 오류가 연쇄적으로 커지는 것을 막을 수 있다.
