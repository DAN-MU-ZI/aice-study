## KPT 회고

### Keep (좋았던 것)
- 목적이 짧고 선명하다. AI 에이전트 로그를 분석해 KPT 회고를 만든다는 한 줄 목표가 분명해서 스킬의 존재 이유가 바로 이해된다.
- 출력 포맷 계약이 좋다. 요약, 세션별 활동, KPT, 타임라인이 모두 정의돼 있어 결과물 품질을 일정하게 유지하기 쉽다.
- `retros/TIMELINE.md`에 누적하는 방식은 단발성 회고가 아니라 프로젝트 학습 이력을 남긴다는 점에서 방향이 맞다.

### Problem (나빴던 것)
- Claude 로그 위치는 구체적이지만 Codex 로그 탐색 규칙은 `유사한 구조` 수준이라 구현자가 흔들릴 여지가 크다.
- JSONL 스키마를 강하게 가정한다. `human`, `tool_use`, `tool_result` 외 변형 포맷이나 누락 케이스에 대한 fallback 규칙이 없다.
- `매 push마다 자동 실행`이라고 적혀 있지만, 실제 트리거가 git hook인지 CI인지 수동 실행인지 계약이 없다.
- 저장 후 바로 `git add`와 커밋 메시지까지 적어둔 흐름은 공격적이다. dry-run, 중복 방지, 사용자 확인 같은 안전장치가 빠져 있다.

### Try (개선해야 할 것)
- 로그 탐색 규칙을 툴별로 명시하자. Codex 경로, 파일 패턴, 실패 시 fallback 순서를 문서에 직접 적는 편이 낫다.
- 실행 모드를 분리하자. `scan only` → `generate retro` → `update timeline` → `stage` 순서로 나누면 자동화와 수동 개입의 경계가 분명해진다.
- `TIMELINE.md` 갱신 규칙을 추가하자. 최신순 삽입, 중복 링크 방지, 파일이 없을 때 헤더 생성 같은 규약이 필요하다.
- 자동 스테이징은 기본값이 아니라 옵션으로 낮추자. 기본 동작은 회고 파일 생성과 타임라인 갱신까지만 두는 편이 안전하다.
