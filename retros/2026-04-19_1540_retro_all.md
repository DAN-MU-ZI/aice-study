### Keep (좋았던 것)
- 초반에는 데이터셋별 실습 세트를 반복 생성하면서 디렉터리 구조, 노트북 쌍, 검증 스크립트라는 기본 산출물 패턴을 빠르게 굳혔다. 덕분에 이후에는 “새 데이터셋에도 같은 골격을 적용한다”는 작업 단위가 분명해졌다.
- 중반 이후에는 샘플 문항과 생성 결과의 차이를 구체적으로 드러내는 방향으로 프롬프트가 발전했다. “난이도가 높다” 수준을 넘어서, 문항 리듬, 배점 구조, 한글 시험체, 블록 세트, 가이드 문구까지 기준이 선명해진 점이 좋았다.
- Jupyter MCP를 본격적으로 활용한 뒤에는 노트북을 텍스트 파일처럼 뭉뚱그려 다루지 않고 셀 단위로 검토·수정하는 흐름이 자리 잡았다. 이는 포맷 정합성과 검수 근거를 동시에 확보하는 방식이었다.
- 후반에는 validator를 “실행 통과 도구”에서 “형식과 적절성 검사기”로 다시 정의하려는 움직임이 있었다. 생성 파이프라인이 성숙해지려면 꼭 필요한 방향 전환이었다.

### Problem (나빴던 것)
- 가장 큰 반복 문제는 환경 전제 불일치였다. `python` 직접 실행, Docker 권한, 네트워크 다운로드, Jupyter RTC, Git lock 같은 문제가 여러 날에 걸쳐 재발했고, 실제 생성/검토 작업보다 운영성 복구에 시간이 많이 들었다.
- 하나의 장문 세션에서 세부 수정이 너무 많이 왕복되면서 프롬프트 수가 비정상적으로 커진 구간이 있었다. 특히 4월 16~17일의 노트북 직접 편집 구간은 의도는 일관됐지만 회차 분리나 중간 산출물 고정이 부족해 재작업 비용이 커졌다.
- skill, README, validator, 생성 코드가 참조하는 경로와 원칙이 완전히 동기화되지 않았다. 예를 들어 `Docker-first`, `솔루션 선행`, `specs 제거`, `한글 시험체` 같은 합의가 일부 지점에는 반영됐지만 다른 지점에는 늦게 반영됐다.
- 후반부로 갈수록 “생성”과 “검증”의 목적이 섞였다. 실제 실행 검증이 필요한지, 구조/형식 검증만 필요한지, 샘플 정합성 비교가 우선인지가 세션마다 다시 정의되며 validator 설계가 흔들렸다.

### Try (개선해야 할 것)
- 생성기 진입 전 공통 preflight를 만들자. `python` 사용 금지 여부, Docker 사용 가능 여부, 네트워크 필요성, 데이터 경로 준비 상태, Git 잠금 상태를 먼저 점검하고, 하나라도 안 맞으면 바로 중단해야 한다.
- 긴 노트북 편집 작업은 단계형 체크포인트를 강제하자. 예를 들면 `front matter`, `데이터 분석`, `전처리`, `모델링`, `딥러닝`, `submission/마감 문구` 순서로 확정하고 각 단계가 끝날 때 diff 또는 셀 목록을 고정하는 편이 낫다.
- skill/README/validator/generator에서 공유하는 규칙을 최소 핵심 원칙으로 묶자. 최소한 `한글 시험체`, `30/30/40`, `14문항`, `솔루션 선행`, `문제 노트북 정답 누출 금지`, `Docker-first`, `format-first validation`은 단일 출처를 가져야 한다.
- validator를 두 층으로 분리하자. `format check`는 빠르고 환경 독립적으로, `execution/artifact check`는 Docker 전제 하에서 선택적으로 실행하게 해야 현재의 혼선을 줄일 수 있다.
